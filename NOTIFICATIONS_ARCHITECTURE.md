# Архитектура внедрения уведомлений в SCADA Mobile

Дата: 2026-03-17

## 1. Цель документа

Цель: описать, как архитектурно грамотно, корректно и безопасно внедрить уведомления в SCADA Mobile так, чтобы при обнаружении ошибки аппарата backend доставлял уведомление всем пользователям приложения.

Ключевое требование для вашего проекта:

- когда приложение открыто, пользователь должен мгновенно видеть проблему внутри интерфейса;
- когда приложение свернуто или закрыто, пользователь должен получить системное уведомление Android;
- Android-обёртка через Bubblewrap не должна содержать бизнес-логики и не должна становиться отдельным источником уведомлений;
- система не должна дублировать уведомления на каждом poll-цикле и не должна замедлять polling PrintSrv.

Этот документ опирается на текущую реализацию backend и frontend и не предлагает ломать уже существующую live-архитектуру.

## 2. Что уже есть в проекте

### 2.1 Backend

В текущем backend уже реализован правильный базовый поток обнаружения инцидента:

1. PrintSrv poller получает новые snapshots.
2. StatusBroadcaster реагирует на PrintSrvInstancePolledEvent.
3. UnitDetailService.extractActiveErrors извлекает активные ошибки аппарата.
4. UnitErrorStore становится единым источником правды по активным ошибкам.
5. AlertService строит AlertMessageDTO для конкретного аппарата.
6. ActiveAlertStore хранит активные алёрты и вычисляет дельту появление/исчезновение.
7. LiveWsHandler рассылает ALERT всем подключённым клиентам через /ws/live.

Итог: backend уже умеет правильно определять момент появления и момент снятия ошибки без повторной рассылки на каждом polling-проходе.

Это очень сильная база. Для push-уведомлений не нужно придумывать новый детектор инцидентов. Нужно только правильно добавить новый канал доставки поверх уже существующего слоя ALERT.

### 2.2 Frontend

Во frontend уже есть зрелая live-модель:

- одно постоянное WebSocket-соединение /ws/live;
- AppContext хранит alerts как глобальную карту активных ошибок;
- useLiveWs умеет принимать ALERT_SNAPSHOT, ALERT и UNITS_STATUS;
- при открытом приложении интерфейс уже может сразу показать проблему пользователю;
- service worker уже зарегистрирован, но пока реализует только install/activate/fetch и не умеет push/notificationclick;
- есть PWA manifest и TWA-конфигурация Bubblewrap.

Итог: in-app уведомления фактически уже существуют как live-обновление интерфейса. Не хватает системных push-уведомлений для фона и закрытого приложения.

### 2.3 Android / Bubblewrap / TWA

Для Android-обёртки в проекте уже есть важные предпосылки:

- приложение разворачивается как TWA поверх PWA;
- используется один и тот же web-клиент, а не отдельный Android UI;
- в twa-manifest.json уже включено enableNotifications: true;
- service worker и manifest уже присутствуют в web-приложении.

Это означает, что для Android не нужно внедрять Firebase SDK в Java/Kotlin-слой и не нужно строить второй канал уведомлений внутри native-обёртки. Правильный путь для вашего стека: Web Push через service worker, который внутри Android TWA будет работать через браузерный стек Chrome.

## 3. Ключевой архитектурный вывод

Для SCADA Mobile нужен не один канал уведомлений, а два канала с разной ответственностью:

### Канал 1. In-app live notifications

Назначение: мгновенно обновлять интерфейс, пока приложение открыто.

Транспорт: существующий WebSocket /ws/live.

Источник истины: существующий AlertService + ActiveAlertStore.

Этот канал уже есть и его нужно сохранить как основной для активной сессии.

### Канал 2. Out-of-app push notifications

Назначение: доставлять системные уведомления, когда приложение свернуто, находится в фоне или закрыто.

Транспорт: Web Push через service worker.

Источник истины: тот же самый ALERT-доменный слой, что и для WebSocket.

Этот канал нужно добавить, но не параллельно бизнес-логике, а как ещё один delivery adapter на том же событии ALERT.

## 4. Почему нельзя делать push прямо из текущего StatusBroadcaster

Наивный вариант выглядит так: внутри StatusBroadcaster, сразу после sendAlert, синхронно отправлять Web Push всем подпискам.

Так делать не стоит по четырём причинам:

1. StatusBroadcaster сейчас находится на горячем пути после polling события. Сетевые вызовы к push endpoint-ам будут замедлять обработку scan cycle.
2. Web Push имеет сетевые ошибки, таймауты и удаление невалидных подписок. Это отдельный жизненный цикл, который нельзя смешивать с обработкой данных оборудования.
3. Если push-рассылка временно недоступна, это не должно блокировать live-обновления UI и не должно влиять на снимки состояния.
4. При росте числа пользователей fan-out по всем подпискам станет отдельной инфраструктурной задачей.

Правильная архитектура: разделить обнаружение инцидента и доставку уведомлений.

## 5. Рекомендуемая целевая архитектура

## 5.1 Общая схема

1. Backend обнаруживает изменение состояния алёрта аппарата.
2. Формируется каноническое внутреннее событие AlertActivated или AlertResolved.
3. Событие сохраняется в outbox или передаётся в отдельный notification orchestrator.
4. Дальше fan-out идёт по двум независимым адаптерам:
   - LiveWsNotifier: отправляет ALERT в /ws/live;
   - WebPushNotifier: отправляет push всем сохранённым подпискам.
5. Frontend при открытом приложении получает ALERT по WebSocket и обновляет UI.
6. Service Worker при фоне или закрытом приложении получает push и показывает системное уведомление.
7. По нажатию на системное уведомление приложение открывается сразу на нужном маршруте аппарата.

## 5.2 Рекомендованный внутренний контракт события

Нужен внутренний notification event, отделённый от транспортного DTO. Например:

- alertId
- alertType
- state: ACTIVATED | RESOLVED
- workshopId
- unitId
- unitName
- severity
- occurredAt
- errorSignature
- errors[]
- route

Где:

- alertId — стабильный идентификатор уведомления для дедупликации и закрытия нотификации;
- errorSignature — хэш нормализованного набора ошибок, чтобы понимать, изменился ли состав ошибки, а не только факт её наличия;
- route — готовый путь вида /workshops/{workshopId}/units/{unitId}/logs или /batch.

## 6. Важное ограничение текущего backend, которое нужно учесть

Сейчас ActiveAlertStore.updateAndDiff определяет только два состояния по unitId:

- ошибки не было -> ошибка появилась;
- ошибка была -> ошибка исчезла.

Если аппарат остаётся аварийным, но список ошибок внутри него изменился, текущая реализация не считает это новой дельтой и не отправляет новый ALERT.

Это означает:

- для сценария один аппарат вошёл в ошибку и должен один раз уведомить всех пользователей текущая архитектура уже подходит;
- для сценария внутри активной аварии поменялась конкретная причина, и это тоже надо донести как новое уведомление текущей логики недостаточно.

Поэтому нужно принять одно из двух решений заранее.

### Вариант A. Семантика один активный алёрт на аппарат

Уведомление отправляется только при первом входе аппарата в аварийное состояние и, опционально, при снятии аварии.

Плюсы:

- минимальные изменения;
- нет спама при нестабильном оборудовании;
- полностью совместимо с текущим ActiveAlertStore.

Минусы:

- если у аппарата во время уже активной аварии изменилась конкретная причина, отдельное уведомление не придёт.

### Вариант B. Семантика уведомлять о смене причины ошибки

Нужно расширить дельта-логику: сравнивать не только наличие активного алёрта, но и errorSignature.

Плюсы:

- пользователи получают уведомление именно об изменившейся причине;
- выше точность бизнес-смысла.

Минусы:

- выше риск шума;
- потребуется нормализация и стабилизация errorSignature, иначе будут ложные дубли.

Для производства я рекомендую начать с варианта A, а вариант B вводить только если операционно подтвердится потребность уведомлять именно о смене причины внутри уже активной аварии.

## 7. Backend: как внедрять правильно

## 7.1 Новый слой ответственности

В backend лучше добавить отдельный notification subsystem, а не размазывать push по существующим ws-классам.

Рекомендуемая структура ответственности:

- AlertDetectionLayer
  источник: текущие AlertService + ActiveAlertStore;
- NotificationEventFactory
  переводит AlertMessageDTO в внутреннее событие уведомления;
- PushSubscriptionRegistry
  хранит и управляет подписками устройств;
- NotificationOutbox
  сохраняет задания на доставку;
- WebPushDeliveryWorker
  отправляет push асинхронно и удаляет мёртвые подписки;
- LiveWsNotifier
  оставляет текущую live-рассылку без блокировок.

## 7.2 Где именно встраивать точку публикации

Публиковать внутреннее notification event нужно не на уровне polling клиента и не на уровне PrintSrv integration, а после вычисления дельты алёрта.

Практически правильная точка: сразу после того, как backend понял, что delta.added или delta.removed не пусты.

То есть архитектурный смысл такой:

- AlertService отвечает за определение проблемы;
- ActiveAlertStore отвечает за переход состояния;
- Notification subsystem отвечает за доставку в разные каналы.

## 7.3 Обязательная асинхронность

Push-доставка должна быть асинхронной относительно polling-пайплайна.

Надёжный вариант для вашего проекта:

- backend сохраняет notification job в БД;
- отдельный worker периодически выбирает pending jobs и отправляет push;
- успешные доставки помечаются sent;
- просроченные или невалидные подписки помечаются inactive;
- scan cycle не ждёт завершения push-рассылки.

Если брокера сообщений в проекте пока нет, лучший production-вариант здесь именно outbox в БД, а не in-memory очередь.

## 7.4 Почему здесь нужна БД

Для push-подписок in-memory storage нельзя считать корректным решением.

Причины:

- после рестарта backend все подписки пропадут;
- несколько инстансов backend не смогут делить подписки между собой;
- невозможно безопасно чистить истёкшие записи и вести аудит;
- невозможно гарантировать доставку через outbox.

Поэтому для push subsystem нужна долговременная persistence.

Минимально достаточные сущности:

### Таблица push_subscriptions

- id
- installation_id
- endpoint
- endpoint_hash
- p256dh_key
- auth_key
- user_agent
- platform
- app_channel
- created_at
- updated_at
- last_seen_at
- active
- notification_permission
- preferred_workshop_id nullable
- preferred_unit_id nullable

Примечания:

- installation_id нужен, потому что в проекте пока нет авторизации, а модель приложения ближе к телефон = сотрудник;
- endpoint_hash полезен для безопасной дедупликации и поиска без хранения endpoint в индексах;
- active позволяет не удалять запись физически сразу после ошибки доставки.

### Таблица notification_outbox

- id
- alert_id
- event_type
- payload_json
- created_at
- next_attempt_at
- attempts
- status: PENDING | SENT | FAILED | DEAD
- last_error

### Таблица notification_delivery_log опционально

- id
- outbox_id
- subscription_id
- delivered_at
- result_code
- error_class

Эта таблица полезна, если потребуется расследование потерь доставки на производстве.

## 7.5 API для подписок

Нужны отдельные REST endpoints, не связанные с /ws/live:

### POST /api/v1.0.0/push/subscriptions

Назначение: upsert подписки устройства.

Тело запроса должно содержать:

- installationId
- subscription.endpoint
- subscription.keys.p256dh
- subscription.keys.auth
- appChannel: PWA | TWA
- platform: android | desktop | unknown
- preferredWorkshopId optional
- preferredUnitId optional

### DELETE /api/v1.0.0/push/subscriptions

Назначение: удалить или деактивировать подписку при revoke permission, logout-подобном действии или смене устройства.

### PATCH /api/v1.0.0/push/subscriptions/preferences

Опционально. Нужен, если позже вы захотите не просто broadcast всем, а управлять подпиской по цеху, аппарату или роли.

Сейчас, по вашему требованию, можно начинать с broadcast всем активным подпискам.

## 7.6 Формат payload для push

Лучший вариант: использовать тот же семантический payload, что и ALERT, но дополнить метаданными доставки.

Пример логического payload:

```json
{
  "type": "ALERT",
  "alertId": "apparatniy:hassia2:critical:2026-03-17T10:12:14",
  "state": "ACTIVATED",
  "workshopId": "apparatniy",
  "unitId": "hassia2",
  "unitName": "HASSIA №2",
  "severity": "Critical",
  "timestamp": "2026-03-17T10:12:14Z",
  "route": "/workshops/apparatniy/units/hassia2/logs",
  "errors": [
    {
      "device": "Printer11",
      "code": 0,
      "message": "Нет связи с принтером"
    }
  ]
}
```

Рекомендации:

- alertId должен быть стабильным для одного активного инцидента;
- route должен строиться на backend, чтобы service worker не пытался угадать навигацию;
- payload не должен включать чувствительные данные, которых не должно быть в системной шторке Android.

## 7.7 Политика отправки

Для production безопаснее такая политика:

- ACTIVATED отправлять всегда;
- RESOLVED отправлять опционально и тише, чем ACTIVATED;
- повторную отправку того же alertId не делать;
- если subscription возвращает 404 или 410, сразу деактивировать её;
- временные сетевые ошибки ретраить с backoff;
- общее число попыток ограничивать;
- при массовой аварии группировать нагрузку батчами.

## 8. Frontend и PWA: как внедрять правильно

## 8.1 Где запрашивать разрешение

Нельзя показывать Notification.requestPermission() автоматически при старте приложения.

Правильно:

- спрашивать разрешение после явного пользовательского действия;
- лучше всего сделать это в настройках или в явном экране Включить системные уведомления;
- до запроса объяснить пользователю, зачем уведомления нужны: аварии аппаратов, остановка линии, снятие аварии.

Это особенно важно для производственного приложения: если разрешение запросить слишком рано, пользователь просто запретит уведомления и больше не вернётся к этому.

## 8.2 Регистрация подписки

Правильный фронтовый поток:

1. Убедиться, что service worker зарегистрирован и активен.
2. Проверить Notification.permission.
3. Получить registration.pushManager.getSubscription().
4. Если подписки нет, выполнить subscribe c VAPID public key.
5. Сгенерировать или прочитать installationId из local storage.
6. Отправить подписку в backend через POST /push/subscriptions.
7. Периодически делать re-sync при старте приложения и при смене подписки.

Важно: installationId должен жить дольше одной сессии и переживать перезапуск приложения. Это заменяет userId, которого сейчас нет в системе.

## 8.3 Что должен делать service worker

Сейчас service-worker.js работает только как offline/cache shell. Для push-архитектуры он должен получить ещё три обязанности:

### push

При получении push:

- распарсить payload;
- проверить, есть ли открытые visible-клиенты этого origin;
- если хотя бы один клиент видим, не показывать системную нотификацию, а при необходимости передать payload в окно через postMessage;
- если видимого клиента нет, показать showNotification.

Это обязательная часть, иначе пользователь будет получать дубли: и UI уже красный, и Android-шторка сверху при открытом приложении.

### notificationclick

При клике по уведомлению:

- найти уже открытую вкладку/окно приложения;
- если клиент найден, сфокусировать его и перевести на route из payload;
- если клиента нет, открыть новый client window на нужном route.

Для вашего роутинга маршрут должен вести не просто в корень, а сразу в аппарат:

- /workshops/{workshopId}/units/{unitId}/logs если нужно открыть журнал ошибки;
- или /workshops/{workshopId}/units/{unitId}/batch если нужен базовый экран аппарата.

### pushsubscriptionchange

Если браузер перевыпустил подписку, service worker должен инициировать повторную регистрацию в backend. Это снижает вероятность тихой потери уведомлений.

## 8.4 Поведение React-приложения при открытом UI

При открытом приложении React уже получает ALERT через /ws/live. Это поведение нужно оставить главным.

Рекомендуемое разделение:

- WebSocket отвечает за мгновенное обновление экрана;
- Service Worker отвечает только за фоновые системные уведомления;
- если push пришёл при видимом окне, service worker не показывает showNotification.

Это убирает конфликт двух каналов и делает UX предсказуемым.

## 9. TWA / Android-специфика

Для вашего стека правильная модель такая:

- backend отправляет стандартный Web Push;
- Chrome внутри Android обрабатывает push;
- service worker решает, показывать ли Android notification;
- Bubblewrap/TWA только контейнеризует PWA и не содержит логики маршрутизации уведомлений.

Это соответствует вашему архитектурному принципу Android-обёртка без бизнес-логики.

### Обязательные условия для стабильной работы на Android

1. HTTPS на production origin.
2. Корректный manifest.webmanifest.
3. Корректный service worker на том же origin.
4. Корректный assetlinks.json и валидная связка сайта с Android package.
5. Устройство должно открывать приложение именно как TWA/PWA, а не случайную вкладку браузера вне доверенного origin.

### Что не нужно делать

- не нужно подключать Firebase SDK в Android-слой только ради push;
- не нужно делать отдельный push backend специально для native APK;
- не нужно дублировать бизнес-правила в Java/Kotlin.

## 10. Безопасность и эксплуатационная надёжность

Для производственной системы уведомления нельзя внедрять только как функциональность. Их надо внедрять как эксплуатационный контур.

## 10.1 Что обязательно с точки зрения безопасности

- VAPID private key хранить только во внешней конфигурации или secret storage;
- никогда не логировать p256dh, auth и endpoint целиком на info-уровне;
- ограничить CORS только доверенными origin;
- валидировать входящий subscription payload;
- защищать endpoint регистрации подписки от мусорных запросов rate limiting-ом;
- не включать в payload чувствительные внутренние данные, которых не должно быть в Android notification preview.

## 10.2 Что обязательно с точки зрения устойчивости

- не слать push синхронно из polling pipeline;
- не слать push на каждом scan cycle;
- делать дедупликацию по alertId;
- удалять мёртвые подписки автоматически;
- отделять временные ошибки доставки от постоянных;
- иметь метрики sent, failed, expired, suppressed_visible_client;
- логировать correlationId или alertId для расследования.

## 10.3 Что обязательно с точки зрения анти-шума

Для производственного контура это критично. Если приложение начнёт спамить, сотрудники отключат уведомления.

Поэтому рекомендую сразу зафиксировать правила:

- один push на один вход аппарата в аварийное состояние;
- повтор того же alertId запрещён;
- resolved-уведомления либо отключены, либо менее агрессивны;
- при flapping-поведении оборудования использовать cooldown, например не более одного нового push по тому же аппарату за окно N секунд, если alert быстро падает и поднимается;
- текст уведомления должен быть коротким и однозначным: аппарат, цех, 1 главная причина.

## 11. Broadcast всем пользователям: как делать безопасно

По вашему требованию уведомление должно уходить всем пользователям приложения.

Архитектурно это допустимо, но в production лучше различать два уровня:

### Этап 1. Глобальный broadcast всем активным подпискам

Подходит для запуска, когда задача проста: любая критическая ошибка аппарата видна всем.

### Этап 2. Управляемый broadcast по контексту

Следующий зрелый шаг, который стоит предусмотреть в модели данных, даже если не включать сразу:

- по цеху;
- по аппарату;
- по роли;
- по моему аппарату;
- по типу инцидента.

Причина простая: на производстве всем полезно знать о критической остановке, но не всем полезно получать каждую локальную техническую ошибку каждой линии.

Поэтому даже если на старте вы рассылаете всем, лучше сразу заложить поля preferredWorkshopId, preferredUnitId и notification scope, чтобы потом не переделывать контракт хранения подписок.

## 12. Рекомендуемая дорожная карта внедрения

## Этап 1. Не ломая текущую архитектуру

- сохранить текущий /ws/live как основной live-канал;
- добавить persistence для push subscriptions;
- добавить service worker push и notificationclick;
- добавить REST endpoints регистрации подписки;
- добавить асинхронный Web Push sender;
- отправлять push только на ACTIVATED.

Это уже даст нужный вам результат: backend увидел ошибку аппарата -> все пользователи приложения получили уведомление, даже если приложение не открыто.

## Этап 2. Повысить эксплуатационную зрелость

- добавить outbox delivery worker с retry и cleanup;
- добавить метрики и алерты на сам контур уведомлений;
- добавить suppress policy для visible clients;
- добавить installationId deduplication.

## Этап 3. Повысить точность доменной семантики

- при необходимости расширить ActiveAlertStore до сравнения errorSignature;
- добавить scoped delivery вместо broadcast всем;
- добавить resolved notifications или silent badge updates.

## 13. Финальная рекомендация

Для вашего проекта архитектурно правильное решение выглядит так:

- не заменять текущие WebSocket ALERT уведомления, а использовать их как live-канал для открытого приложения;
- добавить второй канал доставки через Web Push для фона и закрытого приложения;
- не внедрять логику уведомлений в Android-обёртку Bubblewrap;
- не отправлять push прямо из polling-пайплайна;
- строить push на той же доменной дельте ALERT, которая уже вычисляется backend;
- хранить подписки и outbox в долговременном хранилище, а не в памяти;
- дедуплицировать уведомления по стабильному alertId;
- подавлять системную нотификацию, если приложение уже открыто и видимо пользователю.

Именно эта схема соответствует текущей архитектуре SCADA Mobile, не дублирует бизнес-логику между слоями и даёт безопасный путь к production в связке React PWA + Bubblewrap TWA + Spring backend.
