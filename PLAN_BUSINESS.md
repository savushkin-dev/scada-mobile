# Система оперативного оповещения о простоях — План работ

| Дата | Событие |
|---|---|
| **24 февраля** | Старт работ (текущий момент) |
| **20 марта** | Завершение практики — система полностью работает и готова к показу |
| **середина–конец апреля** | Внедрение: приложение установлено и используется реальными сотрудниками |
| **начало июня** | Сдача дипломной работы |

---

## Текущее состояние (что уже сделано)

| Часть системы | Статус | Что готово |
|---|---|---|
| **Серверная часть** | ✅ Работает | Сервер запущен, получает данные с оборудования, предоставляет информацию о состоянии всех участков, умеет отправлять команды управления, корректно восстанавливается после сбоев связи |
| **Веб-интерфейс** | ⚠️ Прототип | Визуальный прототип готов — есть понимание итогового вида. Полноценный интерфейс для сотрудников пока в разработке |
| **Мобильное приложение (Android)** | ✅ Работает | Собрано и готово к установке на телефоны сотрудников |

**Текущее ограничение:** система пока настроена на одну производственную линию. Для работы с несколькими линиями одновременно требуется доработка, запланированная на ближайшее время.

---

## Фаза 1 — 24 февраля → 20 марта (завершение практики)

Цель: система полностью готова и может быть продемонстрирована руководству.

---

### 24 февр – 5 марта — Поддержка нескольких линий и мгновенные уведомления

- [ ] Настроить систему для работы с несколькими производственными линиями и участками одновременно (не только с одной)
- [ ] Обеспечить мгновенную доставку информации о состоянии оборудования на экраны сотрудников — без ручного обновления страницы
- [ ] Настроить автоматические уведомления при возникновении проблемы на участке: сотрудники получают сигнал немедленно
- [ ] Настроить автоматическое закрытие уведомления, когда проблема устранена
- [ ] Обновить документацию по интерфейсу системы

**Результат:** сервер отслеживает все линии и мгновенно информирует всех заинтересованных сотрудников о любых отклонениях.

---

### 6 марта – 13 марта — Полноценный рабочий интерфейс для сотрудников

- [ ] Разработать удобный веб-интерфейс с отображением всех линий и участков
- [ ] Реализовать переключение между участками производства в один клик
- [ ] Карточка каждого участка: текущий статус, выполняемое задание, счётчик продукции, наличие проблемы (выделено цветом)
- [ ] Цветовой индикатор состояния: работает / простой / ошибка
- [ ] Возможность отправить команду на участок прямо из интерфейса
- [ ] Интерфейс адаптирован для смартфона — сотрудник пользуется в цеху, возможно в перчатках
- [ ] Тёмная тема — учёт условий освещения в цеху

**Результат:** сотрудники видят актуальное состояние всех участков в реальном времени со своего телефона.

---

### 14 марта – 20 марта — Финальная отладка и подготовка к демонстрации

- [ ] Понятное поведение при потере связи с оборудованием: показывается последнее известное состояние с указанием времени и предупреждением
- [ ] Постоянные всплывающие предупреждения об активных проблемах на участках:
  - Появляются мгновенно при возникновении инцидента
  - **Не закрываются** по таймеру и не убираются вручную — исчезают только когда проблема устранена
  - Если проблем несколько — отображается весь список, каждое уведомление живёт независимо
  - Визуально заметны: красный цвет, значок, вибрация телефона
  - Опционально: уведомление приходит даже когда приложение свёрнуто
- [ ] Обновить и проверить мобильное приложение для Android
- [ ] Финальная проверка всей цепочки: оборудование → сервер → интерфейс → телефон

**Результат:** система работает от начала до конца, мобильное приложение готово, практика закрыта.

---

## Фаза 2 — 21 марта → ~20 апреля (путь к внедрению)

Цель: система разворачивается на серверах предприятия, сотрудники начинают пользоваться в штатном режиме.

---

### 21 марта – 31 марта — Мониторинг работоспособности системы

- [ ] Организовать постоянное автоматическое наблюдение за состоянием системы: скорость опроса оборудования, качество связи, очередь команд, актуальность данных, количество подключённых пользователей
- [ ] Настроить наглядный дашборд (панель мониторинга) для ИТ-службы и ответственных специалистов: состояние связи с оборудованием, история инцидентов, активные пользователи
- [ ] Подготовить систему к развёртыванию на серверах компании — упаковать все компоненты для простой установки одной командой

**Результат:** вся система разворачивается одной командой; ИТ-служба видит состояние системы в режиме реального времени.

---

### 1 апреля – 10 апреля — Промышленное развёртывание и автоматизация обновлений

- [ ] Подготовить инфраструктуру для запуска системы в серверной среде предприятия
- [ ] Настроить автоматическую проверку работоспособности системы: при любом обновлении кода автоматически запускаются проверки и система собирается заново
- [ ] Все настройки подключения к оборудованию и серверам — через конфигурационные параметры, без правки кода

**Результат:** система готова к промышленной эксплуатации; обновления проходят без ручного вмешательства.

---

### 11 апреля – 20 апреля — Запуск в эксплуатацию на предприятии

- [ ] Развернуть систему в сети предприятия
- [ ] Подключить к реальному оборудованию на производственных линиях
- [ ] Установить мобильное приложение на телефоны сотрудников
- [ ] Настроить автоматические предупреждения для ИТ-службы при потере связи с оборудованием
- [ ] Подготовить инструкцию для системного администратора предприятия: как обслуживать, обновлять, просматривать журнал событий
- [ ] Финальная проверка в рабочих условиях: все участки отображаются, команды выполняются

**Результат:** приложение работает у реальных сотрудников предприятия.

---

## Фаза 3 — май → начало июня (дипломная работа)

Цель: оформление письменной части диплома; программная часть к этому моменту уже не изменяется (или минимально).

- Оформить архитектурное описание системы с наглядными схемами
- Описать полный жизненный цикл инцидента: от сигнала на оборудовании до уведомления сотрудника
- Описать организацию мониторинга системы
- Описать инфраструктуру развёртывания
- Оформить по требованиям университета

## Ключевые риски

| Риск | Вероятность | Как снижаем |
|---|---|---|
| Нет доступа к оборудованию для тестирования в период разработки | Средняя | Используем тестовую имитацию реального оборудования с заготовленными данными |
| При переезде сайта на новый адрес придётся пересобрать мобильное приложение | Высокая | Обновить конфигурацию и пересобрать APK — задача решаемая, запланирована |
| Задержка с предоставлением доступа к серверам предприятия | Средняя | Тестируем на локальной связке до последнего момента, деплой — последний шаг |
| Оборудование не поддерживает несколько одновременных подключений | Низкая | Проверим при тестировании; при необходимости перестроим схему опроса |
