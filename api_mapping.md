# Сопоставление полей API и MarkPrintServer

В данном документе описано точное соответствие полей JSON, передаваемых между бэкендом (Spring Boot) и фронтендом мобильного приложения, и оригинальных свойств объектов в `Br1_MarkServer` (на основе документации `DATA_FOR_MOBILE.md`).

---

## Архитектура транспортных каналов

Приложение использует **два типа транспорта** одновременно, каждый со своей зоной ответственности:

| Канал | Эндпоинт | Назначение | Жизненный цикл |
|-------|----------|------------|----------------|
| **REST** | `GET /api/v1.0.0/workshops/topology` | Статическая топология цехов (id, name, totalUnits). Ответ содержит `ETag`. | Один раз при старте приложения. При повторном входе — из памяти, без запроса. |
| **REST** | `GET /api/v1.0.0/workshops/{id}/units/topology` | Статическая топология аппаратов цеха (id, workshopId, unit). Ответ содержит `ETag`. | Один раз при первом входе в конкретный цех. При повторном — из памяти. |
| **REST** | `GET /api/v1.0.0/health/live` | Liveness probe — приложение запущено | По необходимости (мониторинг) |
| **REST** | `GET /api/v1.0.0/health/ready` | Readiness probe — получен хотя бы один snapshot | По необходимости (мониторинг) |
| **WebSocket** | `ws://backend/ws/live` | Единый двунаправленный канал: алёрты (все экраны) + live-статус аппаратов (экран цеха). Клиент управляет подпиской через JSON-сообщения. | Открывается при старте приложения, живёт всё время. Один коннект на всё приложение. |
| **WebSocket** | `ws://backend/ws/unit/{unitId}` | Потоковые данные экрана деталей (все 4 вкладки) | ⚙️ *Планируется.* Открывается при входе на экран аппарата, закрывается при уходе. |

### Почему именно так

- **Топология (экраны 1–2)** отделена от live-данных: цехи и аппараты меняются крайне редко (переименование раз в несколько лет). Загружается один раз и хранится в памяти клиента на всё время сессии. Следующий шаг — поддержка `If-None-Match` / `304 Not Modified` для ревалидации кэша без передачи тела ответа.

- **Единый WebSocket `/ws/live`** заменяет три ранее планировавшихся канала (`ws/workshops/status`, `ws/workshops/{id}/units/status`, `ws/alerts`). Одно TCP-соединение исключает дублирование хэндшейков и упрощает обработку переподключений. Клиент сам управляет подпиской — отправляет `SUBSCRIBE_WORKSHOP` при входе в цех и `UNSUBSCRIBE_WORKSHOP` при выходе. Сервер мгновенно шлёт `ALERT_SNAPSHOT` при коннекте, затем пушит `UNITS_STATUS` и `ALERT`-дельты.

- **`problemUnits` для дашборда** вычисляется полностью на клиенте: количество активных алёртов (любого уровня — `Critical` и `Warning`) по каждому цеху из глобального стейта `alerts`. Отдельного WS-канала для счётчиков цехов больше нет.

- **Экран 3 (детали аппарата)** — данные меняются непрерывно: счётчики камер, статус принтера, очередь, активные ошибки. Один WebSocket-поток на аппарат устраняет мигание при перерисовке: бэкенд пушит обновление ровно в момент изменения, фронт применяет патч в store.

## Раздел 1 — REST: `GET /api/v1.0.0/workshops/topology` (Топология цехов)

> **Транспорт:** HTTP GET.
> **Когда вызывается:** один раз при старте приложения. При повторном входе на дашборд данные берутся из памяти — запрос не отправляется.
> **Кэширование (следующий шаг):** ответ содержит заголовок `ETag: "sha256"`. После реализации поддержки `If-None-Match` клиент будет получать `304 Not Modified` при неизменной конфигурации.

Возвращает статическую структуру всех цехов. Не содержит live-полей.

| Поле в JSON API | Описание |
|-----------------|----------|
| `id` | Уникальный идентификатор цеха (например, `apparatniy`, `cmp`) |
| `name` | Название цеха (например, `Аппаратный цех`) |
| `totalUnits` | Общее количество аппаратов/линий в цеху |

> **Важно:** `problemUnits` в этом ответе **отсутствует** — это вычисляемое поле. Клиент самостоятельно подсчитывает количество активных алёртов (`Critical` + `Warning`) по каждому `workshopId` из глобального стейта `alerts`, который наполняется через `/ws/live`.

---

## Раздел 2 — REST: `GET /api/v1.0.0/workshops/{id}/units/topology` (Топология аппаратов цеха)

> **Транспорт:** HTTP GET.
> **Когда вызывается:** один раз при первом входе в конкретный цех. При повторном — из памяти.
> **Кэширование (следующий шаг):** тот же `ETag` что и для `/workshops/topology` (хэш общей конфигурации).

Возвращает статическую структуру аппаратов/линий для указанного цеха. Ответ `404`, если цех не найден.

| Поле в JSON API | Описание |
|-----------------|----------|
| `id` | Уникальный идентификатор аппарата/линии |
| `workshopId` | ID цеха-владельца |
| `unit` | Название аппарата/линии (например, `Линия розлива ПЭТ №1`) |

> **Важно:** `event` и `timer` в этом ответе **отсутствуют** — это live-поля. Они поступают через сообщение `UNITS_STATUS` канала `/ws/live` (после подписки через `SUBSCRIBE_WORKSHOP`). Карточки аппаратов первоначально отображаются со значениями-заглушками (`"Нет данных"`, `"00:00:00"`); live-данные приходят с первым WS-пушем.

## Раздел 2а — REST: Health probes

### `GET /api/v1.0.0/health/live` — Liveness probe

Проверяет, что приложение запущено.

| Поле ответа | Тип | Описание |
|------------|-----|----------|
| `status` | `"UP"` \| `"DOWN"` | Статус жизнеспособности |
| `timestamp` | ISO 8601 | Время проверки (UTC) |

### `GET /api/v1.0.0/health/ready` — Readiness probe

Проверяет готовность: хотя бы один snapshot получен от PrintSrv. Возвращает `200 OK` если готово, `503 Service Unavailable` если нет.

| Поле ответа | Тип | Описание |
|------------|-----|----------|
| `status` | `"UP"` \| `"DOWN"` | Статус готовности |
| `timestamp` | ISO 8601 | Время проверки (UTC) |
| `ready` | `boolean` | `true` если бэк готов принимать трафик |

## Раздел 3 — WebSocket: `ws://backend/ws/live` (Единый канал)

> **Транспорт:** WebSocket, двунаправленный.
> **Жизненный цикл:** открывается при старте приложения, живёт всё время. Одно TCP-соединение на всё приложение.
>
> Канал заменяет три ранее существовавших WS-эндпоинта: `ws/workshops/status`, `ws/workshops/{id}/units/status` и `ws/alerts`.
>
> На момент подключения бэкенд немедленно отправляет снапшот всех активных алёртов (`ALERT_SNAPSHOT`). Далее клиент управляет подпиской на live-статус аппаратов цеха через служебные JSON-сообщения.

---

### Управляющие сообщения: Клиент → Сервер

#### `SUBSCRIBE_WORKSHOP` — подписаться на аппараты цеха

Отправляется при входе пользователя на экран цеха. Сервер начинает пушить `UNITS_STATUS` только для этого цеха. Предыдущая подписка (если была) автоматически снимается — активна ровно одна подписка на сессию.

```json
{
  "type": "SUBSCRIBE_WORKSHOP",
  "workshopId": "apparatniy"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `"SUBSCRIBE_WORKSHOP"` | Константа — команда подписки |
| `workshopId` | `string` | ID цеха, на аппараты которого нужна подписка |

#### `UNSUBSCRIBE_WORKSHOP` — отписаться от аппаратов цеха

Отправляется при уходе с экрана цеха. Сервер прекращает пушить `UNITS_STATUS` для данной сессии.

```json
{
  "type": "UNSUBSCRIBE_WORKSHOP"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `"UNSUBSCRIBE_WORKSHOP"` | Константа — команда отписки |

---

### Серверные сообщения: Сервер → Клиент

#### `ALERT_SNAPSHOT` — начальный снапшот алёртов (при подключении)

Отправляется **сразу при открытии WS-соединения**. Содержит все алёрты, которые сейчас активны. Клиент заменяет (а не патчит) весь стейт `alerts`.

```json
{
  "type": "ALERT_SNAPSHOT",
  "payload": [
    {
      "type": "ALERT",
      "workshopId": "apparatniy",
      "unitId": "hassia2",
      "unitName": "Линия розлива ПЭТ №2",
      "severity": "Critical",
      "active": true,
      "errors": [
        { "device": "CamAgregation", "code": 5, "message": "Дублирование DataMatrix-кода" }
      ],
      "timestamp": "2026-03-01T10:23:45"
    },
    {
      "type": "ALERT",
      "workshopId": "cmp",
      "unitId": "line3",
      "unitName": "Линия №3",
      "severity": "Warning",
      "active": true,
      "errors": [
        { "device": "Line", "code": 0, "message": "Линия остановлена" }
      ],
      "timestamp": "2026-03-01T10:20:00"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `"ALERT_SNAPSHOT"` | Константа — признак начального снапшота |
| `payload` | `AlertMessage[]` | Массив активных алёртов. Может быть пустым (`[]`), если сейчас нет тревог. Структура каждого элемента — та же, что и в `ALERT` (см. ниже) |

#### `UNITS_STATUS` — live-статус аппаратов цеха

Пушится после каждого scan cycle **только** тем сессиям, которые подписаны через `SUBSCRIBE_WORKSHOP`. Содержит текущее состояние всех аппаратов подписанного цеха.

```json
{
  "type": "UNITS_STATUS",
  "workshopId": "apparatniy",
  "payload": [
    { "unitId": "hassia2", "workshopId": "apparatniy", "event": "Активен розлив", "timer": "00:12:45" },
    { "unitId": "hassia3", "workshopId": "apparatniy", "event": "Стоп — нет заготовок", "timer": "00:03:11" }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `"UNITS_STATUS"` | Константа — маршрутизация на клиенте |
| `workshopId` | `string` | ID цеха (подтверждение принадлежности) |
| `payload[].unitId` | `string` | ID аппарата |
| `payload[].workshopId` | `string` | ID цеха-владельца аппарата |
| `payload[].event` | `string` | Текущее событие / описание состояния (из PrintSrv) |
| `payload[].timer` | `string` | Время текущего состояния в формате `HH:MM:SS` |

Клиент патчит только `event` и `timer` поверх топологии аппаратов, загруженной через REST — без повторных HTTP-запросов.

#### `ALERT` — дельта алёртов (при изменении состояния)

Отправляется **всем** подключённым сессиям **только при изменении** состояния алёрта — не каждый scan cycle, а только дельту (появился / исчез). Это — **единственный источник правды** о динамике тревог. Фронт хранит актуальный набор в `Map<unitId, Alert>`.

```json
{
  "type": "ALERT",
  "workshopId": "apparatniy",
  "unitId": "hassia2",
  "unitName": "Линия розлива ПЭТ №2",
  "severity": "Critical",
  "active": true,
  "errors": [
    {
      "device": "CamAgregation",
      "code": 5,
      "message": "Дублирование DataMatrix-кода"
    }
  ],
  "timestamp": "2026-03-01T10:23:45"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | `"ALERT"` | Константа — тип сообщения |
| `workshopId` | `string` | ID цеха (нужен для пересчёта `problemUnits` на дашборде) |
| `unitId` | `string` | ID аппарата (ключ в стейте алёртов) |
| `unitName` | `string` | Читаемое название аппарата (используется в тексте уведомления) |
| `severity` | `"Warning"` \| `"Critical"` | Уровень критичности, формируется бэкендом. Фронт **не вычисляет** уровень самостоятельно |
| `active` | `boolean` | `true` — алёрт появился; `false` — алёрт снят. При `false` фронт удаляет запись из стейта |
| `errors[]` | `array` | Список конкретных ошибок, вызвавших алёрт. При `active: false` может быть пустым |
| `errors[].device` | `string` | Имя устройства-источника (`CamAgregation`, `Printer11`, `Line` и т.д.) |
| `errors[].code` | `integer` | Числовой код ошибки из протокола PrintSrv (при `Warning` всегда `0`) |
| `errors[].message` | `string` | Текстовое описание (из `ErrorMessage`, справочника кодов или `"Линия остановлена"` для Warning) |
| `timestamp` | `ISO 8601` | Время события на стороне бэкенда |

### Логика бэкенда при формировании алёртов

Бэкенд после каждого scan cycle вычисляет текущее состояние алёртов и сравнивает с предыдущим (`ActiveAlertStore`). Алёрт отправляется только при изменении.

**Приоритет при определении severity (в `AlertService`):**

| Условие | Severity | Описание |
|---------|----------|----------|
| `Line.error ≠ 0` или `Line.ErrorMessage ≠ ""` | `Critical` | Активная ошибка устройства |
| Ошибок нет, но `Line.ST ≠ "1"` | `Warning` | Линия остановлена без зафиксированной ошибки |
| Ошибок нет и `Line.ST == "1"` | *(нет алёрта)* | Линия работает нормально |

Бэкенд **не знает** ничего про цвета, вкладки, вибрацию и push-уведомления. Он отдаёт факт: алёрт есть/нет, уровень, детали.

## Раздел 4 — WebSocket: `ws://backend/ws/unit/{unitId}` (Детали аппарата) ⚙️ Планируется

> **Транспорт:** WebSocket, постоянное соединение.
> **Жизненный цикл:** открывается при переходе на экран деталей аппарата, закрывается при нажатии «назад». Бэкенд рассылает обновления после каждого scan cycle только подключённым клиентам данного `unitId`.
>
> Все сообщения — JSON с обязательным полем `"type"`, по которому фронт маршрутизирует обновление в нужный слой стейта (и, соответственно, в нужную вкладку таббара).

### Конверт каждого сообщения

```json
{
  "type": "LINE_STATUS" | "DEVICES_STATUS" | "QUEUE" | "ERRORS",
  "unitId": "hassia2",
  "timestamp": "2026-03-01T10:23:45",
  "payload": { ... }
}
```

---

### Тип `LINE_STATUS` → вкладка «Партия»

Соответствует бывшему `GET /api/line/status`. Бэкенд отправляет при каждом изменении любого поля партии или статуса линии.

| Поле в `payload` | Оригинальное свойство MarkPrintServer | Описание |
|-----------------|---------------------------------------|----------|
| `lineName` | Имя файла проекта (например, `Hassia2.mpr`) | Название линии |
| `lineState` | `Line.ST` | Состояние линии (`1` — работа, `0` — стоп) |
| `shortCode` | `BatchQueue.kmc` / `Printer11.kmc` | Краткий код продукта (КМЦ) |
| `kms` | `BatchQueue.kmc` / `Printer11.kmc` | Код производителя (КМЦ) |
| `description` | `Printer11.descr` / `BatchQueue.description` | Описание продукта |
| `ean13` | `Printer11.ean13` / `BatchQueue.ean13` | Штрихкод EAN-13 |
| `batchNumber` | `Printer11.partynumber` / `BatchQueue.batch` | Номер партии |
| `dateProduced` | `Printer11.dateproduced` / `BatchQueue.dateproduced` | Дата выработки |
| `datePacking` | `Printer11.datepack` / `BatchQueue.datepack` | Дата фасовки |
| `dateExpiration` | `Printer11.dateexpiration` / `BatchQueue.dateexpiration` | Дата годности |
| `initialCounter` | `Printer11.curitem` (парсинг первого значения) | Начальный счётчик (кол-во маркировок) |
| `site` | `Printer11.place` / `BatchQueue.place` | Площадка / Место на линии |
| `itf` | `Printer11.itf` / `BatchQueue.itf` | Код ITF-14 |
| `capacity` | `Printer11.emk` / `BatchQueue.emk` | Ёмкость (количество единиц в коробе) |
| `boxCount` | `Printer11.kole` / `BatchQueue.kole` | Количество коробок / единиц |
| `packageCount` | `Printer11.kolm` / `BatchQueue.kolm` | Количество упаковок / маркировок |
| `freeze` | `BatchQueue.frozen` | Заморозка (`1` — заморожено) |
| `region` | `BatchQueue.region` | Код региона |
| `design` | `Printer11.designe` / `BatchQueue.designe` | ID дизайна этикетки |
| `printDM` | `Printer11.printdm` / `BatchQueue.printdm` | Печать DataMatrix (`1` / `0`) |

### Тип `DEVICES_STATUS` → вкладка «Устройства»

Соответствует бывшему `GET /api/devices/status`. Бэкенд отправляет при изменении состояния любого устройства или счётчиков камер.

| Поле в `payload` | Оригинальное свойство MarkPrintServer | Описание |
|-----------------|---------------------------------------|----------|
| `printer.state` | `Printer11.ST` | Состояние принтера (`1` — работа, `0` — стоп) |
| `printer.error` | `scada.LineDev011Error` / `Printer11.Error` | Наличие ошибки на принтере |
| `printer.batch` | `Printer11.curitem` | Текущая позиция на принтере (маркировка \| партия \| дата) |
| `cam41.read` | `scada.Dev041CounterGeneral` / `CamAgregation.BatchSucceeded` | Количество успешно считанных кодов (Камера 41) |
| `cam41.unread` | `scada.Dev041CounterMissing` / `CamAgregation.BatchFailed` | Количество несчитанных/ошибочных кодов (Камера 41) |
| `cam41.state` | `scada.Dev041Work` / `CamAgregation.ST` | Состояние камеры 41 |
| `cam41.error` | `scada.Dev041Error` | Наличие ошибки на камере 41 (агрегированная) |
| `cam41.batch` | `CamAgregation.curitem` | Текущая позиция на камере 41 (счетчик \| партия \| дата) |
| `cam42.read` | `scada.Dev042CounterGeneral` / `CamAgregationBox.BatchSucceeded` | Количество успешно считанных кодов (Камера 42) |
| `cam42.unread` | `scada.Dev042CounterMissing` / `CamAgregationBox.BatchFailed` | Количество несчитанных/ошибочных кодов (Камера 42) |
| `cam42.state` | `scada.Dev042Work` / `CamAgregationBox.ST` | Состояние камеры 42 |
| `cam42.error` | `scada.Dev042Error` | Наличие ошибки на камере 42 (агрегированная) |

### Тип `QUEUE` → вкладка «Очередь»

Соответствует бывшему `GET /api/queue`. Бэкенд отправляет при любом изменении очереди партий.

| Поле в `payload` | Оригинальное свойство MarkPrintServer | Описание |
|-----------------|---------------------------------------|----------|
| `items[].position` | Индекс `BatchQueue.Item01`...`Item10` | Позиция в очереди (1–10) |
| `items[].shortCode` | Парсинг из `BatchQueue.ItemXX` | Краткий код продукта в очереди |
| `items[].batch` | Парсинг из `BatchQueue.ItemXX` | Номер партии в очереди |
| `items[].dateProduced` | Парсинг из `BatchQueue.ItemXX` | Дата выработки для партии в очереди |

### Тип `ERRORS` → вкладка «Журнал»

Соответствует бывшему `GET /api/errors`. Бэкенд отправляет при изменении набора активных ошибок или поступлении новой записи в лог.

| Поле в `payload` | Оригинальное свойство MarkPrintServer | Описание |
|-----------------|---------------------------------------|----------|
| `deviceErrors[].objectName` | Имя объекта (например, `Dev041` / `CamAgregation`) | Идентификатор устройства с ошибкой |
| `deviceErrors[].propertyDesc` | Описание флага (например, `scada.Dev041Dublicate`) | Человекочитаемое описание ошибки |
| `deviceErrors[].value` | Значение флага (`1` или `0`) | Статус ошибки (`1` — активна) |
| `logs[].time` | `sqlite_logs.db` (поле времени) | Время возникновения события |
| `logs[].ackTime` | `sqlite_logs.db` (поле подтверждения) | Время подтверждения события оператором |
| `logs[].group` | `sqlite_logs.db` (группа) | Группа события (`Система`, `Ошибка` и т.д.) |
| `logs[].description` | `sqlite_logs.db` (описание) | Текст события |

## Раздел 5 — Ответственность фронтенда

Всё, что ниже, — это зона фронтенда. Бэкенд об этом не осведомлён.

### При подключении к `/ws/live` (получение `ALERT_SNAPSHOT`)

1. Полностью заменить стейт `alerts`: `Map<unitId, AlertMessage>` — данными из `payload`.
2. Пересчитать `problemUnits` для каждого цеха: `alerts.values().filter(a => a.workshopId === id).length` — считаются все алёрты, независимо от `severity`.
3. Перекрасить карточки цехов на дашборде согласно `severity` (`Critical` → красный, `Warning` → жёлтый).

### При получении `ALERT { active: true }`

1. Записать в глобальный стейт: `alerts.set(unitId, { workshopId, severity, errors, timestamp })`.
2. Обновить карточку цеха на дашборде: пересчитать `problemUnits` по `workshopId` и перекрасить по `severity`.
3. Обновить карточку аппарата в списке: перекрасить по `unitId` и `severity`.
4. Вызвать `navigator.vibrate([200, 100, 200])` (если приложение на экране).
5. Вызвать `self.registration.showNotification(unitName, { body: errors[0].message, vibrate: [200, 100, 200] })` через Service Worker (работает даже когда приложение свёрнуто).

### При получении `ALERT { active: false }`

1. Удалить запись из глобального стейта: `alerts.delete(unitId)`.
2. Вернуть нормальный цвет карточке цеха и карточке аппарата, пересчитать `problemUnits`.
3. Убрать соответствующее уведомление (если реализован стек нотификаций в UI).

### Цветовая семантика severity

| Severity | Цвет карточки | Когда |
|----------|--------------|-------|
| `Critical` | Красный | Активная ошибка устройства (`Line.error ≠ 0` / `ErrorMessage ≠ ""`) |
| `Warning` | Жёлтый | Линия остановлена без зафиксированной ошибки (`ST ≠ "1"`, `error == 0`) |
| *(нет алёрта)* | Нормальный | Линия работает штатно |

### При входе на экран цеха

- Отправить `SUBSCRIBE_WORKSHOP` с нужным `workshopId` через `/ws/live`.
- Получать `UNITS_STATUS` — патчить только `event` и `timer` поверх уже загруженной топологии.

### При уходе с экрана цеха

- Отправить `UNSUBSCRIBE_WORKSHOP` через `/ws/live`.

### При клике на карточку аппарата (экран 2 → экран 3)

```
if (alerts.has(unitId)) → открыть экран деталей на вкладке «Журнал»
else                     → открыть экран деталей на вкладке «Партия»
```

Решение о стартовой вкладке принимается **только на фронтенде** в момент навигации.

### При открытии экрана деталей аппарата

- Установить WebSocket-соединение `ws://backend/ws/unit/{unitId}`.
- Получить первое сообщение каждого типа (`LINE_STATUS`, `DEVICES_STATUS`, `QUEUE`, `ERRORS`) — это первичная гидратация всех вкладок.
- Последующие сообщения применяются как патч к уже отображённым данным — без мигания, без перезапроса.

### Бейдж на иконке вкладки «Журнал»

Количество активных записей `deviceErrors` (с `value = 1`) из последнего сообщения типа `ERRORS`. Обновляется при каждом новом сообщении — пользователь видит счётчик ошибок не переключая вкладки.

## Раздел 6 — Web Push (уведомления когда приложение закрыто) ⚙️ Планируется

> Web Push на Android работает поверх **FCM (Firebase Cloud Messaging)** автоматически через Chrome — никакого FCM SDK подключать не нужно. Используется стандартный браузерный **Web Push Protocol** (VAPID) на бэкенде и **Service Worker** на фронтенде.

### Регистрация подписки

| Шаг | Кто | Действие |
|-----|-----|----------|
| 1 | Фронтенд | `Notification.requestPermission()` — запросить разрешение у пользователя |
| 2 | Фронтенд | `PushManager.subscribe({ applicationServerKey: VAPID_PUBLIC_KEY })` — получить `PushSubscription` |
| 3 | Фронтенд | `POST /api/push/subscribe` — отправить объект подписки на бэкенд |
| 4 | Бэкенд | Сохранить `{ endpoint, keys: { p256dh, auth } }` для каждого клиента |

### Отправка при срабатывании алёрта

Бэкенд при каждом `ALERT` делает **оба действия одновременно** — они не взаимоисключающие:

| Ситуация | WebSocket `/ws/live` | Web Push |
|----------|----------------------|----------|
| Приложение открыто и на экране | ✅ отправить | фронт подавляет системную нотификацию (`document.visibilityState === 'visible'`) |
| Приложение свёрнуто | ✅ отправить | ✅ отправить → Service Worker → шторка Android |
| Приложение закрыто | соединения нет | ✅ отправить → Service Worker будит Chrome → шторка Android |
| WebSocket отвалился (потеря сети) | недоступен | ✅ fallback — уведомление всё равно дойдёт |

### Формат Web Push payload (тот же ALERT)

Бэкенд отправляет Web Push с тем же JSON что и в `/ws/live` (сообщение типа `ALERT`). Service Worker при получении `push`-события показывает нотификацию и при клике открывает приложение сразу на нужном аппарате.

### Что нужно добавить в бэкенд

| Что | Детали |
|-----|--------|
| Зависимость | `nl.martijndwars:web-push` (Java Web Push Protocol) |
| VAPID-ключи | Сгенерировать один раз, хранить в конфиге (`application.yml`) |
| `POST /api/push/subscribe` | Принять и сохранить `PushSubscription` от клиента |
| `DELETE /api/push/subscribe` | Удалить подписку при явном выходе (опционально) |
| При формировании ALERT | Вызвать `PushSender.send(subscription, alertPayload)` для всех сохранённых подписок |
